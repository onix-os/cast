# Plan: Extract a Language-Agnostic Declaration Core

**Status:** Active prerequisite for [`lua.md`](lua.md)
**Priority:** P0
**Effort:** Large
**Risk:** High
**Planned against:** `develop` at `ea3f1036`
**Audit date:** 2026-07-23

This plan must be completed before Lua is connected to any production domain. The purpose is to
separate Cast's declaration infrastructure from Gluon, then reconnect all current Gluon behavior
through the new adapter boundary. When this plan finishes, Cast is still Gluon-only from a user's
point of view, but Gluon is no longer the architecture.

[`lua.md`](lua.md) begins only after the completion criteria here pass. Lua then becomes another
adapter over the same source security, limits, module graph, diagnostics, identity, persistence,
and domain contracts. It must not copy or fork them.

## 1. Desired result

The current shape is effectively:

```text
domain crate -> Gluon-specific DTO/evaluator -> gluon_config -> Gluon VM
config store -> GluonCodec/load_gluon/save_gluon -> .glu files
```

The target shape is:

```text
                         crates/declarative_config
              source security / limits / deadline / diagnostics
               module graph / identity / evaluation result types
                    adapter and persistence contracts
                                  |
                         Gluon implementation only
                                  |
                  crates/gluon_config::GluonEngine
                                  |
          per-domain Gluon evaluator/codec adapters -> shared domain values
                                  |
        stone_recipe / mason / forge / triggers / config manager
```

After this plan:

- `declarative_config` knows no Gluon, Lua, YAML, KDL, or domain ABI.
- `gluon_config` owns Gluon parsing, import extraction, VM construction, interruption, error
  translation, and typed extraction.
- Each domain exposes engine-neutral declaration entry points and semantic values. Its Gluon-only
  structs and derives are isolated inside a Gluon adapter module.
- `config::Manager` loads and persists through language/codec contracts. Only the Gluon
  implementation is registered and `.glu` remains the sole accepted extension.
- Through Phase 6, existing `.glu` sources, versioned ABIs, generated bytes, diagnostics, limits,
  fingerprints, and domain results remain exactly compatible. Phase 7 is the sole deliberate
  identity boundary: authored `.glu` remains valid and historical materialized generations remain
  addressable, but identity-bearing v1 locks/digests may become stale for new derivations.
- Lua can later implement the same contracts without changing the shared core again.

## 2. Non-negotiable constraints

- This plan adds **no Lua dependency, Lua source, `.lua` discovery, or dual-language behavior**.
- YAML and KDL remain removed. No fallback, sniffing, or compatibility parser is introduced.
- `.stone` remains the package artifact. Package construction and transaction logic are untouched.
- Atomic updates, rollback architecture, container-trigger behavior, USR merge compliance, and
  OS/local-configuration separation remain outside the evaluator and must not be redesigned here.
- Do not read or modify `../bedrock`.
- Work on `develop`; leave `main` untouched and do not leave migration branches behind.
- Use the root Makefile for every relevant build, test, and validation path. Put new targets in a
  focused imported fragment such as `misc/make/declaration-tests.mk`; keep the FOL-style root
  Makefile small.
- Commit each green architectural slice. Do not combine core extraction, Gluon reconnection,
  domain migration, and identity migration in one commit.
- Do not change release/version metadata.
- Do not add or restore `.clippy.toml`, `.rustfmt.toml`, or `.typos.toml`.
- Do not use Python to edit files.
- No file may exceed 1,000 lines. Split by behavior with descriptive names before crossing the
  limit; never create numbered chunks.
- Do not wrap Git, Make, or compilation in `timeout`. Bound only commands that execute a built
  evaluator/application and can actually hang.
- No ESP, BOOT, block-device, activation, rollback, reboot, or destructive system operation may
  run on the host. This foundation should need only host-safe temporary-directory tests. If a
  system-state migration fixture needs a real installation, use only the disposable VM and do not
  extend into the deferred reboot/power-loss campaign.
- Unrelated improvements go to `FUTURE_PLAN.md` and do not expand this plan.

### Known baseline validation issue

At this baseline, `make source-loc` fails on the pre-existing 2,374-line `CHANGELOG.md`. It also
wraps the LOC script in `timeout`, and the script/test currently time-limit Git and ordinary helper
commands, contradicting the user's rule that only execution of the built application/evaluator may
be bounded. Resolve both baseline issues separately before implementation and do not mix them into
a declaration-core commit. Every file added or touched by this plan must remain below 1,000 lines
from its first commit.

## 3. Current coupling that must be separated

### 3.1 `gluon_config` currently owns both policy and engine

`crates/gluon_config` is about 3,031 Rust LOC. Its production modules combine reusable behavior
with Gluon implementation details:

| Current file | Reusable responsibility | Gluon-only responsibility |
|---|---|---|
| `src/source.rs` | descriptor-rooted hardened reads, retained root identity, bounds | none in its security model; only ownership/naming is Gluon-coupled |
| `src/limits.rs` | source/import/graph/memory/stack/time policy | current stack meaning is Gluon-specific |
| `src/deadline.rs` | one monotonic total deadline | none |
| `src/diagnostic.rs` | stable error categories and source locations | Gluon error decoding/rendering |
| `src/fingerprint.rs` | canonical source/import/input identity | `gluon_version` field and Gluon-specific hash domain |
| `src/import.rs` | graph bounds and rooted resolution | Gluon AST visitor, `import!`, `.glu` normalization, namespace rules |
| `src/evaluator.rs` | pipeline ordering and panic containment | VM creation, watchdog interrupt, `Getable`/`VmType` extraction |

The core cannot be made agnostic by copying these files into another crate and leaving both copies
alive. There must be one owner for source security, limits, deadlines, graph accounting, and
generic identity.

### 3.2 Configuration storage is named after Gluon

`crates/config` exposes `GluonCodec`, `DecodedGluon`, `load_gluon`, `save_gluon`,
`delete_gluon`, and rooted Gluon helpers. Its fragment enumeration, temporary names, generated-file
marker, and extension validation are hardcoded around `.glu`. The atomic persistence and retained
directory logic are reusable, but they are hidden inside Gluon-named modules.

### 3.3 Domain boundaries decode Gluon directly

There are 12 root declaration shapes:

1. package;
2. build-policy total;
3. build-policy patch;
4. policy-layer manifest;
5. source lock;
6. build lock;
7. profile fragment;
8. repository fragment;
9. system intent/snapshot;
10. boot topology;
11. root-filesystem intent;
12. trigger.

The current tree has 129 `Getable`/`VmType` derive sites, 123 in production. Five crates directly
depend on Gluon/codegen: `gluon_config`, `stone_recipe`, `mason`, `forge`, and `triggers`.
This plan does not need to remove those last four dependencies yet, but it must confine them to
clearly named Gluon adapter modules so shared domain logic has no VM types in its public contract.

### 3.4 Persistent identity is Gluon-specific

`EvaluationFingerprint` stores `gluon_version` and hashes under
`os-tools-gluon-evaluation\0`. That shape reaches derivation provenance, locks, package metadata,
CLI explanations, trigger evidence, and exact boot-intent contracts. It cannot become a neutral
type through a field rename.

Identity work therefore has two checkpoints:

1. **Equivalence checkpoint:** extraction and Gluon reconnection preserve fingerprint v1 bytes and
   all current generated data.
2. **Neutral-identity checkpoint:** after every Gluon consumer uses the adapter, introduce a
   versioned engine-neutral identity and migrate Gluon deliberately. Lua cannot start before this
   checkpoint is complete.

## 4. Stable contracts to create

These are conceptual contracts. The executor may refine Rust signatures during the Phase 0 type
spike, but may not merge their responsibilities or leak a VM type into the shared crate.

### 4.1 Language and engine descriptors

`declarative_config` owns stable scalar descriptions:

- `LanguageId`: declaration language/profile identity; the first adapter instantiates it for Gluon;
- `EngineId`: runtime implementation plus exact version identity;
- `LanguageSpec`: canonical extension, generated-file marker, source profile, and engine identity;
- `AbiId`: semantic module name/version, independent of implementation language;
- `EvaluatorPolicyId`: version for capability, resource, graph, and output rules.

Do not model these as free-form unvalidated strings throughout consumers. Construction must enforce
canonical nonempty names and versions so fingerprints and diagnostics cannot disagree.
The types are validated opaque descriptors instantiated by adapters. The core must not contain a
built-in `Gluon` enum variant, Gluon version, or special Gluon extension branch.

### 4.2 Source and prepared graph

The shared core owns:

- `Source` and `SourceRoot`;
- logical names separate from host paths;
- bounded descriptor-rooted reads;
- explicit input identities;
- `ImportRequest` with embedded-versus-relative kind;
- `PreparedModule`/`PreparedGraph` containing canonical logical identity, source hash, and ordered
  dependency edges;
- graph limits, stable ordering/deduplication, cycle observation/policy, and aggregate byte
  accounting.

The engine adapter owns syntax discovery. For Gluon, its AST visitor extracts `import!` requests
and enforces Gluon namespaces/extensions. The core resolves the returned requests; it never parses
Gluon syntax itself.

Embedded modules come from an adapter/domain-supplied `AbiCatalog`, not a core hardcoded map. Each
entry binds `AbiId`, language/profile, exact immutable source bytes/hash, and any engine-specific
primitive/capability requirements. A generic import policy allowlists ABI IDs; the active adapter
provides only sources for its own language. During P0, `gluon_config` builds this catalog from the
current embedded modules and primitive policy. Lua later supplies a separate catalog for the same
semantic ABI IDs without changing core resolution.

### 4.3 Evaluation adapter

Use an associated/concrete engine type and typed callback boundary; do not force all VM values into
one untyped universal tree merely to obtain object safety. The adapter supplies its validated
`LanguageSpec`, engine-specific prepared type, limit application, `with_source_root` evaluation,
and typed decoder. Writable codecs add canonical `encode` separately; read-only evaluators do not
implement a fake encoder. Avoid `dyn Any` and a mutable global registry.

The minimum engine contract is:

1. return its language/engine descriptors;
2. parse a source and return literal import requests plus engine-prepared state;
3. create a fresh restricted runtime;
4. execute only the core-prepared graph under the shared deadline and limits;
5. invoke an engine-specific typed decoder before the runtime is dropped;
6. translate parse/type/import/limit/runtime failures into the shared diagnostic envelope;
7. return `Evaluation<T>` containing an owned domain value and shared evaluation identity.

`declarative_config` must not depend on `Getable`, `VmType`, Gluon threads, Gluon AST nodes, or
Gluon errors. `gluon_config::GluonEngine` may use all of them behind the boundary.

### 4.4 Domain evaluator and codec contracts

The core exposes two separate generic roles:

- `DeclarationEvaluator<T>`: source to owned semantic/wire value plus identity;
- `DeclarationCodec<T>`: evaluation plus canonical generated encoding when that domain is writable.

Do not require an encoder for read-only domains such as triggers merely to satisfy one oversized
interface. The config manager receives the appropriate typed evaluator/codec rather than knowing
which VM implements it.

Each domain should end with:

```text
shared semantic/domain type and validation
              ^
              |
      shared wire representation
              ^
              |
declaration/gluon.rs implements the typed Gluon conversion/codec
```

Lua will later add `declaration/lua.rs` beside the Gluon adapter and target the same shared wire and
semantic types.

### 4.5 Discovery and generated-authority contracts

The generic config layer must expose all three discovery shapes before Lua begins:

- `RootDeclarationSlot`: one fixed logical basename such as `stone`, `system`, `boot-topology`, or
  `root-filesystem`, resolved across registered language extensions with collision detection;
- `FragmentDeclarationSet`: ordered vendor/admin/user directories, same-layer logical-stem
  collision detection, validation of every discovered fragment, and whole-fragment shadowing;
- `GeneratedDeclarationSlot`: one active codec/path/marker per logical output, retained-directory
  save/delete/update, authored-file protection, and transactional authority switching.

Trigger `tx.d`/`sys.d` directories use the fragment shape even though their domain is read-only.
Recipe locks and system snapshots use generated slots. The contracts are parameterized by opaque
`LanguageSpec` values and have no Gluon/Lua branch. During P0 only Gluon is registered, so only
`.glu` is public.

### 4.6 Diagnostics

The shared diagnostic envelope owns stable categories:

- parse;
- schema/type;
- import;
- I/O;
- resource limit;
- runtime;
- internal.

It also owns logical source identity, optional span/line/column, field path, engine descriptor, and
causal context. Gluon-specific error traversal/rendering remains in `gluon_config`. Existing Gluon
diagnostic text and category tests are characterization evidence; change them only in a separate
reviewed diagnostic commit.

### 4.7 Evaluation identity v2

The neutral identity contains:

- language/profile identity;
- engine kind and exact version;
- configuration ABI and evaluator-policy versions;
- root logical name and source hash;
- canonical prepared-module graph and ABI identities;
- explicit-input identities and hashes;
- resource-policy identity;
- a new neutral hash domain and encoding version.

Equal domain values produced by different engines must not share evaluation identity. ABI semantic
versions do not change merely because an engine adapter changes.

Inventory how v1 evidence is actually stored before designing compatibility. Validate a full
`GluonFingerprintV1` only where the full structure exists. Build locks, package metadata, and other
containers that persist only a scalar digest must classify that digest through their containing
legacy schema. Never deserialize or expand a scalar v1 digest into v2 by filling guessed fields;
re-evaluate an exact retained source graph when a new v2 identity is required.

| Target | Owns | Must not own |
|---|---|---|
| new `crates/declarative_config/` | source/root security, limits, deadline, prepared graph, generic diagnostics, language/engine descriptors, identity v2, adapter/result contracts | Gluon/Lua crates, ASTs, VM errors, domain DTOs, file-discovery policy for a specific language |
| `crates/gluon_config/` | Gluon import syntax/policy, VM construction, watchdog interrupt, stack/memory application, error mapping, v1 compatibility, typed extraction | duplicate source loader, fragment layering, package/repository/system meaning |
| `crates/config/` | typed evaluator/codec orchestration, registered-language discovery, fragment precedence, atomic persistence, retained directories, authored/generated protection | `gluon_config`/Gluon manifest dependencies, VM construction, AST parsing, domain validation |
| `crates/stone_recipe/` | package/policy/lock shared wire and semantic types; Gluon domain adapters | source-root implementation or generic fragment manager |
| `crates/mason/` | recipe/profile/source-lock orchestration and Gluon adapters; CLI presentation | engine construction hidden in generic business logic |
| `crates/forge/` | repository/system/boot domain types and Gluon adapters | generic evaluator/source security |
| `crates/triggers/` | trigger domain type and Gluon evaluator adapter | writable-codec behavior |

Likely new shared modules are `source.rs`, `limits.rs`, `deadline.rs`, `diagnostic.rs`,
`language.rs`, `module_graph.rs`, `identity.rs`, `adapter.rs`, and `evaluation.rs`. Keep each module
focused; do not recreate the current monolith under a new name.

## 6. Implementation phases

Every phase ends with a clean tree, focused Make gate, and cohesive commit. The core is not allowed
to drift ahead of its only reference implementation: Gluon must stay green after every extraction.

### Phase 0 — Freeze the Gluon baseline and prove the type boundary

1. Resolve the pre-existing repository LOC and non-runtime-timeout blockers separately.
2. Inventory current public `gluon_config`, `config`, and domain evaluation APIs.
3. Record golden Gluon evidence for source-root errors, limits, import graphs, diagnostic
   categories, fingerprint v1 bytes, generated files, and all 12 normalized root values.
4. Characterize deadline start/end behavior separately for direct `Evaluator::evaluate_file` and
   fragment loading through `config::Manager`; today they do not cover identical work.
5. Classify every relevant Make test as core policy, Gluon engine, config storage, domain adapter,
   or end-to-end regression.
6. Prototype only the Rust type signatures for `LanguageSpec`, `Evaluation<T>`,
   `DeclarationEvaluator<T>`, and `DeclarationCodec<T>` in an isolated test module. Prove they can
   express read-only triggers and writable fragments without `dyn Any`, global registries, or a
   universal untyped value tree.
7. Add `declarative-config-test`, `gluon-adapter-test`, and `declaration-regression-test` through
   `misc/make/declaration-tests.mk` before production extraction.

**Commit examples:** `test(config): freeze gluon declaration contracts`, then
`test(config): prove declaration adapter types`.

**Exit:** fingerprint v1 and generated-file goldens are committed; the adapter signatures compile
for one read-only and one writable domain; no production code moved and no Lua exists.

### Phase 1 — Extract source authority and hardened reads

1. Create `crates/declarative_config` with workspace-inherited package metadata and dependencies.
2. Move `Source`, `SourceRoot`, descriptor-rooted reads, retained root/directory identity,
   the existing openat2 policy, metadata sandwiches, and size/type checks. Do not invent a fallback
   path that the current loader does not have.
3. Preserve public compatibility re-exports from `gluon_config` while callers move.
4. Move source tests with the implementation and keep Gluon integration tests at the adapter edge.

**Commit:** `refactor(config): extract hardened declaration sources`.

**Exit:** there is one source loader implementation; Gluon source behavior and diagnostics are
unchanged; no consumer needs to know the implementation moved.

### Phase 2 — Extract limits, deadline, and diagnostic envelope

1. Move engine-neutral source/import/graph/memory/time policy into focused shared modules.
2. Keep the 64-KiB Gluon stack application in the Gluon adapter while representing its configured
   value in shared policy. Do not claim other engines can enforce it yet.
3. Move the monotonic deadline-budget primitive without silently standardizing caller coverage.
   Preserve each characterized entry point's current start boundary through the equivalence phase.
4. Create the generic diagnostic envelope and let Gluon translate into it.
5. Preserve exact existing Gluon failure categories and timeout/panic containment.

**Commit examples:** `refactor(config): extract evaluation policy and deadline`, then
`refactor(gluon): adapt diagnostics to declaration core`.

**Exit:** core policy contains no VM calls; Gluon resource and diagnostic goldens remain equal.

### Phase 3 — Separate module discovery from graph resolution

1. Move canonical module identity, graph bounds, ordering/deduplication, aggregate byte accounting,
   and rooted/embedded resolution into `declarative_config`.
2. Keep Gluon's AST visitor, `import!` recognition, forbidden namespaces, optional primitive
   policy, and `.glu` normalization in `gluon_config`.
3. Make the Gluon extractor return typed `ImportRequest` values to the shared resolver.
4. Make each current domain/import policy supply a Gluon `AbiCatalog`; remove embedded-source
   lookup from generic resolution without changing the accepted ABI set or primitive exposure.
5. Preserve current reachable-module deduplication, including current cycle behavior, and prove
   identical prepared graphs and fingerprint-v1 module ordering for all fixtures. Do not add cycle
   rejection during extraction; that requires a later explicit evaluator-policy version.

**Commit:** `refactor(config): separate module graph from gluon imports`.

**Exit:** the shared graph never parses source syntax; the Gluon adapter never opens relative files
outside `SourceRoot`.

### Phase 4 — Run Gluon through the evaluation adapter

1. Implement `GluonEngine` around fresh empty VM construction, embedded modules, memory/stack
   policy, watchdog interruption, panic containment, and typed `Getable` extraction.
2. Make shared orchestration own the stages—root load, import discovery, graph preparation,
   identity inputs, VM call, typed decode, and owned `Evaluation<T>` result—while preserving the
   characterized deadline start boundary of each current entry point.
3. Preserve the current `gluon_config::Evaluator` API temporarily as a thin wrapper.
4. Migrate direct callers to the adapter in small groups, then remove the wrapper after the last
   caller. Do not retain parallel old/new evaluator implementations.
5. Add a test-only `SyntheticEngine` with a distinct opaque language/engine descriptor, prepared
   source type, tiny literal-import parser, deterministic interruption signal, native error type,
   and typed decoder. Run it through the shared source/graph/deadline/diagnostic/evaluation pipeline.
   It is not Lua, has no production registration, and writes no tracked declarations.

**Commit examples:** `refactor(gluon): implement declaration engine adapter`, then one caller-group
commit at a time.

**Exit:** every Gluon evaluation passes through the shared pipeline and produces identical v1
identity, diagnostics, and domain values. `gluon_config` contains no duplicate loader/deadline/graph
implementation, and the second-engine seam is proven without Lua.

### Phase 5 — Generalize all discovery, storage, and generated authority

1. Rename internal `GluonCodec`/`DecodedGluon`/`load_gluon` concepts to typed declaration terms.
2. Move fragment collection and atomic persistence into behavior-named declaration modules.
3. Represent file language through `LanguageSpec`, but register only Gluon and accept only `.glu`.
4. Preserve vendor -> admin -> user precedence, whole-fragment shadowing, fail-closed validation,
   generated markers, temporary naming, retained descriptors, authored-file protection, atomic
   rename, and directory synchronization.
5. Migrate in-tree callers and delete old Gluon-named compatibility APIs once no caller remains.
6. Remove `gluon_config`, `gluon`, and `gluon_codegen` from `crates/config/Cargo.toml`; composition
   code must supply the Gluon evaluator/codec from outside the generic storage crate.
7. Add two test-only synthetic adapters with opaque IDs and non-production extensions. Prove exact
   extension dispatch, no fallback, same-layer same-stem collision, cross-layer shadowing while
   every discovered fragment is still validated, and one generated authority per logical slot.
   These adapters must never be registered in production or create tracked declaration files.
8. Migrate every non-fragment discovery path to the P0 contracts before Lua begins:
   - recipe roots and generated source/build lock slots;
   - rooted transaction/system trigger directories;
   - fixed system, boot-topology, and root-filesystem intent slots;
   - generated system snapshots.
   Only Gluon is registered, so public paths remain unchanged.

Suggested final paths:

- `crates/config/src/declaration.rs`;
- `crates/config/src/declaration/fragment_collection.rs`;
- `crates/config/src/declaration/atomic_persistence.rs`;
- `crates/config/src/rooted_declaration.rs`.

**Commit examples:** `refactor(config): generalize declaration persistence`, then
`refactor(config): remove gluon-named storage API`.

**Exit:** user-visible filenames and bytes remain `.glu`; storage code depends on contracts, not
the Gluon VM; its manifest has no Gluon dependency; the synthetic second-adapter seam is proven;
no old public compatibility layer remains.

### Phase 6 — Put all domain semantics behind typed adapters

Migrate by domain and commit each green slice:

1. triggers, proving the read-only evaluator interface;
2. profile and repository fragments, proving writable codecs and layering;
3. package plus four builders;
4. build-policy total/patch and policy layers;
5. source lock and build lock;
6. system intent/snapshot;
7. boot topology and root-filesystem intent.

For each domain:

- separate shared wire/semantic types and validation from Gluon VM structs;
- confine `Getable`, `VmType`, and Gluon optional/boolean wrappers to
  `declaration/gluon.rs` or an equally explicit adapter module;
- implement `DeclarationEvaluator<T>` and, only when writable, `DeclarationCodec<T>`;
- preserve ABI semantic name/version and canonical Gluon emitter bytes;
- expose engine-neutral business entry points; remove `evaluate_gluon` naming after all in-tree
  callers migrate;
- run domain and end-to-end regression gates before moving to the next domain.

Split `crates/mason/src/source_lock.rs` by decoding, canonical Gluon emission, and domain validation
before extending the adapter structure; it is already near the line limit.

**Commit by domain**, for example `refactor(triggers): use declaration evaluator`,
`refactor(repository): isolate gluon codec`, and `refactor(recipe): isolate package adapter`.

**Exit:** all 12 shapes enter shared domain validation through typed contracts. General planners,
stores, and CLI orchestration do not import Gluon VM types.

### Phase 7 — Introduce neutral identity v2 through Gluon

Do this only after the equivalence checkpoint in Phases 1–6 is green.

1. Add the v2 schema from section 4.7 beside the exact v1 type and validator.
2. Make Gluon produce both identities in a characterization window; prove v1 remains byte-identical
   while v2 is deterministic and includes the Gluon language/engine descriptors.
3. Migrate derivation provenance, locks, package metadata, CLI explanations, triggers, and exact
   system/boot source contracts to v2 in separate commits.
4. Treat v1 evidence as stale for new derivations after activation, never as malformed v2. Where a
   legacy container stores only a scalar evaluation digest, classify it through that container's
   legacy schema rather than pretending a full fingerprint is available.
5. Validate full v1 structures only where the inventory proves they exist. Re-evaluate exact
   retained source graphs to produce v2; never fabricate missing engine/policy fields.
6. Preserve immutable historical generations and prove rollback selection/export does not rewrite
   them or require their declaration identity to masquerade as v2.
7. Activate one shared evaluator-policy rule before completion:
   - each root or fragment receives one budget starting immediately before its descriptor-rooted
     open/read and ending after typed decode;
   - shadowed fragments retain separate budgets and are still evaluated;
   - graph preparation rejects import cycles with a stable import diagnostic before VM execution.
   Treat both changes from characterized Gluon behavior as an explicit policy-version bump with
   before/after tests; do not hide them inside extraction commits.
8. Document expected cache/derivation and evaluator-policy invalidation before activation.

**Commit examples:** `feat(config): add engine-neutral evaluation identity`, then consumer-specific
migration commits, then `refactor(config): make gluon identity v2 authoritative`.

**Exit:** all new Gluon evaluations and derivations use v2; full v1 evidence and legacy scalar
digests are handled only according to their actual containing schemas; there is no Lua and no
declaration-language file-format change.

### Phase 8 — Close the foundation and publish the adapter contract

1. Delete temporary re-exports, dual-pipeline scaffolding, and old Gluon-named generic APIs.
2. Verify workspace dependencies: only explicit Gluon adapter modules may import Gluon/codegen.
3. Add compile-time and dependency tests proving `declarative_config` and `config` do not depend on
   `gluon_config`, `gluon`, or `gluon_codegen`.
4. Include `declarative_config` and adapter source trees in semantic implementation fingerprints.
5. Update architecture/configuration documentation to say Gluon is the only registered adapter,
   not the core architecture.
6. Run the complete Make matrix and record the accepted commit.

**Commit examples:** `refactor(config): remove declaration migration shims`, then
`docs(config): document the gluon adapter boundary`.

**Exit:** the foundation completion criteria below all pass. Only then may `lua.md` begin.

Required focused targets, defined in `misc/make/declaration-tests.mk` and imported by the root
Makefile:

- `make declarative-config-test` — shared source, limits, deadline, graph, diagnostics, identity,
  and the test-only synthetic engine;
- `make gluon-adapter-test` — Gluon AST/import, VM, limits, diagnostics, v1/v2, typed extraction;
- `make declaration-regression-test` — all 12 domains plus config storage through Gluon;
- existing domain-specific hardening/example gates;
- `make build`;
- `make test`;
- `make check`;
- `make examples`;
- `make source-loc` after its pre-existing blocker is separately resolved.

The equivalence checkpoint must compare:

- normalized owned Rust values for all 12 root shapes;
- parse/type/import/I/O/limit/runtime/internal error categories and logical source names;
- source/import/graph limit boundaries;
- prepared module identities and ordering;
- fingerprint v1 bytes before Phase 7;
- canonical `.glu` output for draft, profile, repository, system snapshot, source lock, and build

- fragment precedence, invalid-shadow behavior, atomic save/delete, and authored-file protection;
- repeated-process determinism.

This plan does not require real disk or boot mutation. Any system-intent tests must stop at owned
Rust models/plans or use established non-destructive fixtures.

## 8. Completion criteria

This plan is complete only when:

- the known repository LOC and non-runtime-timeout baseline blockers are resolved before the first
  implementation phase;
- `crates/declarative_config` has no Gluon/Lua/domain dependency;
- there is exactly one source loader, deadline implementation, graph resolver, generic diagnostic
  envelope, and v2 identity implementation;
- the generic resolver consumes adapter/domain-supplied `AbiCatalog` entries and contains no
  embedded Gluon source or primitive map;
- all production Gluon evaluation enters through `GluonEngine` and typed domain adapters;
- the test-only `SyntheticEngine` proves distinct prepared state, import extraction, interruption,
  native diagnostics, and typed decoding through the unchanged shared pipeline without production
  registration;
- all 12 domain shapes expose engine-neutral entry points and shared semantic validation;
- `config::Manager` operates on declaration evaluator/codec contracts while registering only
  Gluon and accepting only `.glu`; recipe roots/locks, trigger directories, fixed system/boot/root

## 10. Stop conditions

Stop and report evidence if:

- the shared crate would need a Gluon/Lua VM, AST, error, or DTO dependency;
- the proposed trait requires an unbounded universal value tree, `dyn Any`, or global mutable
  adapter registry to support typed domains;
- source security, deadline coverage, graph bounds, or diagnostics would be duplicated;
- a Gluon value/fingerprint/generated-file golden changes before the explicit v2 phase;
- identity v2 could accept v1 data by guessing missing fields;
- historical generations would need mutation or could no longer be exported/selected;
- config storage begins accepting any extension other than `.glu` during this plan;
- the abstraction requires touching package transaction/activation logic;
- a file reaches 1,000 lines without a functional split;
- validation requires host disk, ESP, BOOT, activation, rollback, or reboot effects;
- completion requires `../bedrock`, version metadata, YAML/KDL, Nix translation, Lua work, or
  formatting/typo-policy configuration;
- an unrelated improvement appears—record it in `FUTURE_PLAN.md`.
